# Requisitos de Seguranca RA3 - EducaFlix

Este arquivo contem exatamente 10 requisitos de seguranca alinhados ao codigo atual do projeto. Ele pode ser exportado para PDF antes da entrega final.

| ID | Categoria | Origem | Requisito | Justificativa | Status | Local no codigo | Como demonstrar na defesa | Teste relacionado |
| -- | --------- | ------ | --------- | ------------- | ------ | --------------- | ------------------------- | ----------------- |
| RS-01 | DFD/STRIDE - Spoofing | Analise de autenticacao | O sistema deve autenticar usuarios antes de permitir acesso a funcionalidades protegidas. | Reduz risco de uso indevido de contas e funcoes privadas. | Implementado | `backend/src/middlewares/authMiddleware.js`; `backend/src/routes/index.js` | Acessar `GET /api/me` sem cookie e mostrar HTTP 401; depois fazer login e acessar com cookie. | `backend/tests/security.test.js` |
| RS-02 | DFD/STRIDE - Tampering | Analise de manipulacao de dados | O sistema deve validar dados recebidos antes de criar ou alterar videos e avaliacoes. | Impede persistencia de dados incompletos, fora de faixa ou com formato invalido. | Implementado | `backend/src/validators/schemas.js`; `backend/src/middlewares/validateRequest.js`; `backend/src/routes/index.js` | Tentar criar video com URL invalida ou duracao zero e mostrar HTTP 400. | `backend/tests/security.test.js` |
| RS-03 | DFD/STRIDE - Information Disclosure | Analise de exposicao de dados | O sistema nao deve retornar senha, hash ou salt nas respostas da API. | Evita vazamento de credenciais mesmo quando endpoints de cadastro, login ou admin retornam usuarios. | Implementado | `backend/src/repositories/userRepository.js`; `backend/src/controllers/authController.js`; `backend/src/controllers/adminController.js` | Fazer cadastro/login e mostrar que a resposta possui usuario sem `senhaHash` e sem `senhaSalt`. | `backend/tests/security.test.js` |
| RS-04 | Codificacao segura | OWASP Password Storage Cheat Sheet | O sistema deve armazenar senhas com hash criptografico e salt, nunca em texto puro. | Protege senhas em caso de acesso indevido ao arquivo de persistencia. | Implementado | `backend/src/services/passwordService.js`; `backend/src/controllers/authController.js`; `backend/src/database/sqliteDatabase.js` | Abrir o banco `backend/data/database.sqlite` apos cadastro e comparar que `senhaHash` difere da senha original pela camada de leitura da aplicacao. | `backend/tests/security.test.js` |
| RS-05 | Codificacao segura | OWASP ASVS | O sistema deve proteger rotas privadas com autenticacao, autorizacao e CSRF em operacoes mutaveis. | Reduz acesso indevido e requisicoes forjadas em rotas com alteracao de estado. | Implementado | `backend/src/middlewares/authMiddleware.js`; `backend/src/middlewares/adminMiddleware.js`; `backend/src/routes/index.js`; `backend/src/services/sessionService.js` | Mostrar que usuario comum recebe HTTP 403 ao tentar criar video e que POST/PUT/DELETE exigem `X-CSRF-Token`. | `backend/tests/security.test.js` |
| RS-06 | Codificacao segura | CWE-89 / OWASP | O sistema deve evitar injecao em consultas usando consultas parametrizadas na camada de repository. | Reduz risco de injecao e mantém o acesso ao SQLite centralizado no backend. | Implementado | `backend/src/repositories/userRepository.js`; `backend/src/repositories/videoRepository.js`; `backend/src/repositories/reviewRepository.js`; `backend/src/database/sqliteDatabase.js` | Apontar os `prepare(...).run/get/all` com parametros e mostrar que as entradas continuam validadas antes do acesso ao banco. | `backend/tests/security.test.js` |
| RS-07 | Livre | Boas praticas de configuracao | O sistema deve carregar configuracoes sensiveis e de ambiente por variaveis, com exemplo versionado. | Evita segredos reais no repositorio e facilita execucao local/CI. | Implementado | `.env.example`; `backend/.env.example`; `backend/src/middlewares/securityHeaders.js`; `backend/src/database/sqliteDatabase.js`; `.gitignore` | Mostrar `.env.example` e `.gitignore` ignorando `.env` e o arquivo SQLite local. | Nao aplicavel |
| RS-08 | Livre | Validacao de entrada | O sistema deve validar campos obrigatorios no cadastro, login, videos, avaliacoes e status de usuario. | Reduz erros de negocio e entradas inconsistentes antes da camada de persistencia. | Implementado | `backend/src/validators/schemas.js`; `backend/src/middlewares/validateRequest.js` | Enviar payload sem campos obrigatorios ou com senha fraca e mostrar HTTP 400. | `backend/tests/security.test.js` |
| RS-09 | Livre | Controle de acesso por propriedade | O sistema deve impedir que um usuario altere ou remova avaliacao de outro usuario. | Preserva integridade e privacidade de dados vinculados a conta. | Implementado | `backend/src/controllers/reviewController.js`; `backend/src/repositories/reviewRepository.js` | Criar avaliacao com um usuario e tentar alterar com outro, esperando HTTP 403. | `backend/tests/security.test.js` |
| RS-10 | Livre | Qualidade e automacao | O sistema deve possuir testes automatizados para autenticacao, validacao, autorizacao e entidade principal. | Garante regressao minima para os requisitos de seguranca exigidos na RA3. | Implementado | `backend/tests/security.test.js`; `.github/workflows/tests.yml`; `package.json`; `backend/package.json` | Rodar `npm test` localmente e mostrar o workflow `Tests` no GitHub Actions. | `backend/tests/security.test.js` |
